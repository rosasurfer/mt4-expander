# MT4 history API conversion subtasks

This document breaks the native MT4 history API conversion into manageable work packages. It is meant to sit next to the broader conversion plan and to make early implementation iterations smaller, reviewable, and independently testable.

## Subtask 1: API contract inventory

**Goal:** Turn the current MQL-facing history include into an implementation checklist.

**Inputs:**

- `mql40/include/rsf/history.mqh` as the authoritative API/signature contract.
- `mql40/libraries/rsfHistory1.mq4` as the behavior reference.
- Existing C/C++ history structs under `header/struct/mt4/`.

**Deliverables:**

- A function-by-function mapping table from MQL declarations to planned C/C++ exports.
- A return-value and error-behavior checklist for every exported function.
- A list of flags, constants, and default parameters that must remain MQL-compatible.
- Confirmation notes for format-401 string handling and absolute-path support policy.

**Done when:** every MQL history entry point has an owner C/C++ function name, a parameter list, a return contract, and a parity note.

## Subtask 2: Fixture and comparison tooling

**Goal:** Create tools that make parity testing cheap before writing the full implementation.

**Deliverables:**

- A small history-file decoder/dumper that prints headers and decoded bars for both format 400 and format 401.
- Golden `.hst` fixtures generated by the current production MQL implementation:
  - empty/newly-created files,
  - contiguous bars,
  - sparse/gapped histories,
  - weekly and monthly boundary cases,
  - both bar formats 400 and 401.
- A binary/semantic comparison tool:
  - byte-for-byte comparison where deterministic,
  - decoded header/bar comparison where reserved bytes may differ.
- A repeatable temporary-directory layout for native and legacy outputs.

**Done when:** a developer can generate or decode fixtures and compare legacy-vs-native output without opening MetaTrader manually for every check.

## Subtask 3: Native scaffolding and handle registry

**Goal:** Add the minimum C/C++ infrastructure for owning history set/file handles in the DLL.

**Deliverables:**

- `header/lib/history.h` and `src/lib/history.cpp` added to the Visual Studio project.
- Process-global registries for history-set and history-file state.
- Handle allocation, lookup, validation, and close/reset helpers.
- A small synchronization wrapper, for example a `CRITICAL_SECTION`, around global registry access.
- Logging/error helpers that follow the repository's existing DLL export style and avoid exceptions crossing the DLL boundary.

**Done when:** placeholder exports can create, validate, and close native handles without touching `.hst` file contents yet.

## Subtask 4: Path and file I/O helpers

**Goal:** Centralize all filesystem behavior before porting history logic.

**Deliverables:**

- A path resolver for the three MQL modes:
  - empty directory → current trade-server history directory,
  - relative directory → MQL sandbox/files-relative path,
  - absolute directory → either explicitly supported or explicitly rejected according to the chosen policy.
- Directory creation helpers matching legacy behavior for write access.
- Binary file open helpers for read-only, write-only/reset, and read-write modes.
- File size, seek, truncate, flush, and exact read/write helpers.
- A clear choice between CRT `FILE*` and Win32 `HANDLE`; prefer Win32 `HANDLE` if share mode, flush behavior, and large offsets need precise control.

**Done when:** native code can create, open, resize, and close test files with the same missing-file and access-mode return behavior as MQL.

## Subtask 5: Header and bar codec layer

**Goal:** Make format-specific bytes independent from higher-level history algorithms.

**Deliverables:**

- Header creation/validation helpers for `HISTORY_HEADER`.
- Common `double[6]` bar representation helpers using `BAR_T`, `BAR_O`, `BAR_H`, `BAR_L`, `BAR_C`, and `BAR_V` ordering.
- Format 400 encode/decode helpers.
- Format 401 encode/decode helpers.
- Unit or harness checks for exact struct sizes and known fixture bytes.

**Done when:** read/write round trips for one bar in both formats preserve expected timestamps, OHLC, and volume values.

## Subtask 6: Metadata and timeframe helpers

**Goal:** Port the time/range calculations that all higher-level operations depend on.

**Deliverables:**

- Helpers to calculate bar open, close, and next-close times for minute, daily, weekly, and monthly periods.
- Metadata structs for stored range, total range, last stored bar, and buffered bar.
- Metadata initialization from empty and non-empty files.
- Cache invalidation/update rules documented close to the code.
- Focused tests for week starts, month starts, month ends, leap years, and sparse-history boundaries.

**Done when:** opening an existing fixture correctly reconstructs stored/total ranges and cached first/last-bar metadata.

## Subtask 7: Primitive file operations vertical slice

**Goal:** Implement the first useful native API slice without tick aggregation.

**Deliverables:**

- Native open/close/read-bar/write-bar exports matching `mql40/include/rsf/history.mqh`.
- Offset-to-file-position logic: `sizeof(HISTORY_HEADER) + offset * barSize`.
- Header creation for newly-created files.
- Close-time flushing of modified caches where relevant.
- MQL wrapper wiring for only these initial functions.

**Done when:** MQL or a native harness can create a format 400/401 history file, write bars, close it, reopen it, and read the same bars back.

## Subtask 8: Search, update, insert, and move primitives

**Goal:** Port random-access history editing after the vertical slice is proven.

**Deliverables:**

- `FindBar` with metadata fast paths and binary-search fallback.
- `UpdateBar` for OHLCV updates from tick values.
- `InsertBar` for append and middle insertion.
- `MoveBars` implemented with buffered block movement and overlap-safe direction.
- Regression tests for inserting before first bar, between bars, appending after last bar, and moving large blocks.

**Done when:** decoded output matches legacy MQL for sparse and out-of-order bar writes.

## Subtask 9: Tick aggregation and buffering

**Goal:** Port the high-level tick-to-bar logic once primitive editing is reliable.

**Deliverables:**

- `HistoryFile.AddTick` parity for all supported flags:
  - `HST_BUFFER_TICKS`,
  - `HST_SKIP_DUPLICATE_TICKS`,
  - `HST_FILL_GAPS`,
  - `HST_TIME_IS_OPENTIME`.
- Buffered-bar flush on period change and close.
- Gap-fill behavior matching legacy output.
- Duplicate tick handling matching legacy output.
- Performance benchmark for buffered vs unbuffered tick streams.

**Done when:** a deterministic tick stream produces decoded bars equivalent to the legacy MQL implementation for M1, H1, D1, W1, and MN1.

## Subtask 10: History-set orchestration

**Goal:** Implement multi-timeframe symbol management on top of proven file operations.

**Deliverables:**

- `HistorySet.Create`, `HistorySet.Get`, `HistorySet.Close`, and `HistorySet.AddTick` equivalents.
- Lazy open of the nine standard timeframe files.
- Same-symbol/same-directory close/reset behavior matching legacy MQL.
- Tests that add one tick stream to all standard timeframes and verify every output file.
- Stress test that opens more files than the MQL per-module handle limit.

**Done when:** a single symbol history set can be created, filled, closed, reopened, and verified across all standard timeframes.

## Subtask 11: Migration wrapper and compatibility switch

**Goal:** Let MQL callers move to the native implementation safely.

**Deliverables:**

- `mql40/include/rsf/history.mqh` imports/wrappers updated for native exports.
- A compile-time or configuration switch to keep the legacy MQL implementation available during rollout.
- Compatibility aliases only where the authoritative include requires them.
- Documentation for rollout, fallback, and known behavior differences.

**Done when:** existing MQL call sites can run against either the legacy or native implementation without source changes outside the history include/wrapper.

## Subtask 12: End-to-end validation and performance sign-off

**Goal:** Prove the native API is correct, faster, and not constrained by the MQL file-handle limit.

**Deliverables:**

- End-to-end parity run comparing legacy and native outputs for representative tick streams.
- Performance report for one-symbol, many-symbol, buffered, and unbuffered scenarios.
- Handle-limit stress test above 64 simultaneously open files.
- Manual MT4 smoke test with generated history files.
- Final checklist of any intentional differences from legacy behavior.

**Done when:** native output is accepted as equivalent to legacy output for production scenarios and the measured performance improvement is documented.

## Recommended implementation order

1. API contract inventory.
2. Fixture and comparison tooling.
3. Native scaffolding and handle registry.
4. Path and file I/O helpers.
5. Header and bar codec layer.
6. Metadata and timeframe helpers.
7. Primitive file operations vertical slice.
8. Search/update/insert/move primitives.
9. Tick aggregation and buffering.
10. History-set orchestration.
11. Migration wrapper and compatibility switch.
12. End-to-end validation and performance sign-off.

This order intentionally front-loads tooling and byte-level codecs so every later behavior port can be checked against production MQL output in small increments.
